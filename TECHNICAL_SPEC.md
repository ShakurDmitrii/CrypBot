# Техническое описание проекта: сайт и Telegram-бот криптообменника

## 1. Цель проекта

Проект представляет собой сайт и Telegram-бот для криптообменника. Основная задача - дать пользователю понятный путь обмена: выбрать направление, увидеть рассчитанный курс, оставить заявку, пройти необходимые проверки и получить сопровождение через сайт или Telegram.

Сайт должен работать как публичная витрина и точка входа в обмен. Telegram-бот должен дублировать ключевые сценарии сайта: расчет курса, создание заявки, коммуникация с оператором, уведомления по статусу и возможная AML-проверка.

## 2. Основные пользовательские сценарии

1. Пользователь открывает сайт.
2. Видит брендовый интерфейс в стиле "Неба": спокойная визуальная система, облака, анимация обмена, смена времени суток.
3. Выбирает направление обмена: валюта или фиат входит в "облако", на выходе отображается криптовалюта или другая валюта.
4. Сайт получает базовый курс из Rapira.
5. Система применяет собственный процент или маржу обменника.
6. Пользователь видит итоговый курс, сумму к получению и условия.
7. Пользователь подтверждает заявку.
8. Система передает заявку оператору, в Telegram-бот или в админ-панель.
9. Пользователь получает статус заявки на сайте и/или в Telegram.
10. При необходимости выполняется AML-проверка кошелька или транзакции.
11. Пользователь может перейти к публичной оферте и ознакомиться с правилами сервиса.

## 3. Что нужно реализовать

### 3.1. Сайт

Нужно реализовать публичный сайт криптообменника со следующими блоками:

- Главный экран с брендингом "Неба".
- Анимация обмена: "облако принимает валюту - облако выпускает крипту".
- Визуальная смена времени суток: день, вечер, ночь. Режим может зависеть от локального времени пользователя или быть декоративным переключателем.
- Калькулятор обмена.
- Список направлений обмена.
- Отображение итогового курса с учетом собственной комиссии.
- Блок с локацией офиса или точки здания на карте.
- Ссылка на публичную оферту.
- Кнопка или встроенное окно Telegram-бота.
- Блок доверия: AML-проверка, регламент, сроки обработки, поддержка.
- Информация для размещения на BestChange, если сервис будет туда подключаться.

### 3.2. Telegram-бот

Telegram-бот должен выполнять функции операционного помощника:

- Показать актуальный курс.
- Рассчитать сумму обмена.
- Принять заявку.
- Запросить реквизиты.
- Передать заявку оператору.
- Уведомлять пользователя о смене статуса.
- Дать ссылку на публичную оферту.
- При необходимости запускать AML-проверку адреса или транзакции.
- Собирать историю обращений пользователя.

Минимальная версия бота может работать без полноценного личного кабинета, но должна сохранять заявки и статусы.

### 3.3. Административная часть

Для стабильной работы обменника нужна админ-зона или хотя бы операторская панель:

- Список заявок.
- Статусы: новая, ожидает оплату, оплата получена, в обработке, завершена, отменена, спорная.
- Ручное изменение курса или маржи.
- Настройка процента к курсу Rapira.
- Управление направлениями обмена.
- Управление лимитами.
- Просмотр AML-результатов.
- Комментарии оператора.
- Лог действий по заявке.

Если админ-панель не делать в первой версии, временно можно отправлять заявки в закрытый Telegram-чат операторов, но это хуже масштабируется и повышает риск ошибок.

## 4. Интеграции

### 4.1. Rapira

Rapira используется как источник базового курса. Логика расчета:

1. Backend получает базовый курс по выбранной паре.
2. Система применяет собственный процент обменника.
3. Пользователь видит уже итоговый курс.
4. В заявке фиксируется курс на момент создания.

Пример формулы:

```text
Итоговый курс = Курс Rapira * (1 + маржа обменника)
```

Для обратных направлений формула может отличаться:

```text
Итоговый курс = Курс Rapira * (1 - маржа обменника)
```

Точный расчет нужно описать отдельно для каждого направления, потому что для покупки и продажи криптовалюты маржа может применяться по-разному.

Узкие места:

- Нужно проверить актуальную документацию Rapira API.
- Нужно понять, доступны ли публичные курсы без авторизации.
- Нужно определить лимиты запросов.
- Нужно кешировать курсы, чтобы не дергать API на каждый ввод пользователя.
- Нужно хранить курс заявки, чтобы после создания заявки он не менялся задним числом.
- Нужно решить, сколько минут действует зафиксированный курс.

### 4.2. Telegram

Возможны два уровня интеграции:

- Простая интеграция: кнопка "Открыть Telegram" и переход в бот.
- Расширенная интеграция: виджет или окно на сайте, имитирующее чат, с передачей заявки в Telegram-бот.

Важно: Telegram не дает полноценный встроенный веб-чат как обычные live-chat сервисы. Чаще всего сайт открывает Telegram-бота по deep link, например с параметром заявки. Если нужно именно окно чата на сайте, потребуется собственный web-chat интерфейс, который будет связан с backend и операторской частью.

Узкие места:

- Нужно выбрать механику: переход в Telegram или собственный чат на сайте.
- Нужно защитить бота от спама.
- Нужно не хранить чувствительные данные в открытых сообщениях.
- Нужно продумать, как оператор видит заявку и отвечает пользователю.

### 4.3. AML-проверка

AML-проверка нужна для оценки риска кошелька или транзакции. Возможны варианты:

- Подключить внешний AML-сервис с API.
- Использовать уже имеющуюся "общую программу", если у нее есть API или возможность автоматизации.
- Делать проверку вручную оператором на первом этапе.

Что нужно уточнить:

- Какая именно программа AML уже есть.
- Есть ли у нее API.
- Какие сети поддерживаются: BTC, ETH, TRON, BSC, TON и другие.
- Что проверяем: адрес, транзакцию или оба варианта.
- Какой риск считать допустимым.
- Что делать с заявкой при высоком риске.

Узкие места:

- AML-результат не всегда однозначен.
- Разные провайдеры дают разные risk score.
- Нужно хранить результат проверки в заявке.
- Нужен регламент для спорных случаев.
- Нельзя показывать пользователю лишнюю внутреннюю AML-информацию без юридической оценки.

### 4.4. BestChange

BestChange может быть каналом привлечения пользователей. Для размещения там обычно нужно подготовить обменник технически и операционно:

- Стабильные направления обмена.
- Понятные резервы.
- Быстрое обновление курсов.
- Страницы правил и публичной оферты.
- Рабочая поддержка.
- Корректные статусы заявок.
- Надежная обработка спорных ситуаций.

Что нужно реализовать до подключения:

- Страницу с правилами обмена.
- Публичную оферту.
- Автоматический или полуавтоматический расчет курсов.
- Отображение резервов.
- Быструю обработку заявок.
- Логи заявок и действий оператора.
- Техническую готовность отдавать актуальные курсы и резервы, если это потребуется для мониторинга.

Узкие места:

- Нужно изучить актуальные требования BestChange к новым обменникам.
- Потребуются резервы и стабильная операционная работа.
- Нужна репутация и понятная поддержка.
- Ошибки в курсах могут привести к убыткам.
- Ручной режим обработки может быть узким местом при росте трафика.

## 5. Локация офиса и точка здания на карте

На сайте нужно показать локацию конторы:

- Адрес текстом.
- Интерактивная карта.
- Точка здания на карте.
- Возможное описание входа или ориентира.
- Часы работы, если есть офлайн-прием.

Возможные реализации:

- Яндекс Карты.
- Google Maps.
- 2GIS.

Что нужно получить:

- Точный адрес.
- Координаты точки.
- Название компании или вывески.
- Нужен ли маршрут до входа.
- Можно ли публично показывать адрес.

Узкие места:

- Если адрес не должен быть публичным, нужно показывать только город или район.
- Карты требуют API-ключи или имеют ограничения на использование.
- Координаты должны быть точными, иначе пользователи будут приходить не туда.

## 6. Публичная оферта и юридические документы

На сайте обязательно нужна ссылка на публичную оферту. Также желательно подготовить:

- Правила обмена.
- Политику конфиденциальности.
- AML/KYC-политику.
- Пользовательское соглашение.
- Контакты поддержки.
- Условия возврата или отмены заявки.
- Регламент фиксации курса.

Важно: тексты юридических документов должен проверить юрист, особенно если сервис работает с фиатом, криптовалютой, AML/KYC и публичным адресом.

Узкие места:

- Нельзя копировать чужую оферту без адаптации.
- Нужно описать момент фиксации курса.
- Нужно описать ответственность сторон.
- Нужно описать действия при ошибочном платеже.
- Нужно описать AML-отказ и возврат средств.
- Нужно учитывать юрисдикцию, в которой работает компания.

## 7. Рекомендуемая архитектура

### 7.1. Frontend

Назначение:

- Публичный сайт.
- Калькулятор.
- Анимации.
- Карта.
- Форма заявки.
- Переход в Telegram-бот.

Возможный стек:

- Next.js или React для полноценного сайта.
- HTML/CSS/JavaScript для простой первой версии.
- Tailwind CSS или собственная CSS-система.

### 7.2. Backend

Назначение:

- Получать курсы Rapira.
- Применять маржу.
- Создавать заявки.
- Хранить статусы.
- Интегрироваться с Telegram-ботом.
- Интегрироваться с AML-сервисом.
- Отдавать данные для сайта.

Возможный стек:

- Node.js + NestJS/Express.
- PostgreSQL для заявок и пользователей.
- Redis для кеша курсов и rate limit.

### 7.3. Telegram-бот

Назначение:

- Общение с пользователем.
- Создание заявок.
- Уведомления.
- Операторские сценарии.

Возможный стек:

- Node.js + Telegraf.
- Общий backend API с сайтом.

### 7.4. База данных

Минимальные сущности:

- users
- exchange_requests
- exchange_directions
- rates
- aml_checks
- operator_comments
- audit_logs
- settings

Ключевой принцип: все важные действия по заявке должны логироваться.

## 8. Модель заявки

У заявки должны быть следующие поля:

- ID заявки.
- Источник: сайт, Telegram, BestChange.
- Пользователь.
- Telegram ID, если есть.
- Направление обмена.
- Сумма отправки.
- Сумма получения.
- Базовый курс Rapira.
- Маржа обменника.
- Итоговый курс.
- Время фиксации курса.
- Срок действия курса.
- Реквизиты пользователя.
- Реквизиты обменника.
- Статус.
- AML-результат.
- Комментарий оператора.
- Даты создания и обновления.

## 9. Безопасность

Нужно предусмотреть:

- HTTPS.
- Хранение секретов только в переменных окружения.
- Rate limit на формы и API.
- Защиту Telegram-бота от спама.
- Логи действий операторов.
- Разделение ролей операторов и администраторов.
- Резервное копирование базы.
- Маскирование чувствительных данных.
- Защиту админ-панели двухфакторной авторизацией.

Узкие места:

- Ошибки в расчете курса могут стоить денег.
- Утечка токена Telegram-бота даст доступ к коммуникациям.
- Утечка API-ключей AML или карт может привести к расходам и рискам.
- Ручная обработка заявок без логов повышает риск спорных ситуаций.

## 10. Расчет курса и маржи

Нужно заранее определить правила:

- Маржа единая или разная по направлениям.
- Маржа в процентах или фиксированная.
- Есть ли минимальная комиссия.
- Есть ли разные курсы для разных сумм.
- Как часто обновляется курс.
- На сколько минут фиксируется курс.
- Что делать, если Rapira API недоступна.

Рекомендуемая логика:

1. Курсы обновляются по расписанию, например каждые 10-30 секунд.
2. Backend хранит последний успешный курс.
3. При расчете пользователю показывается актуальный курс из кеша.
4. При создании заявки курс фиксируется.
5. Если курс старше допустимого времени, заявка не создается.
6. В админке можно временно отключить направление.

## 13. Главные узкие места проекта

1. Юридическая часть: оферта, AML/KYC, правила возврата, юрисдикция.
2. Точность курсов: ошибка в формуле или API может привести к финансовым потерям.
3. Фиксация курса: нужно четко определить срок действия курса.
4. Ручная обработка: оператор может ошибиться в сумме, реквизитах или статусе.
5. AML: нужно понимать, какой риск допустим и что делать с подозрительными средствами.
6. Telegram: бот удобен, но не должен быть единственным местом хранения заявок.
7. BestChange: потребуется стабильная операционная работа и выполнение требований мониторинга.
8. Безопасность: токены, API-ключи, админка, база заявок.
9. Карты и адрес: публичность адреса нужно согласовать заранее.
10. Масштабирование: при росте трафика ручные процессы станут бутылочным горлышком.

12. Запустить MVP на ограниченный трафик.

## 16. Критерии готовности MVP

MVP можно считать готовым, если:

- Пользователь может рассчитать обмен.
- Курс считается с учетом маржи.
- Заявка создается и сохраняется.
- Оператор видит заявку.
- Пользователь получает статус.
- Есть публичная оферта.
- Есть карта или указание локации.
- Есть базовый AML-процесс.
- Секреты не лежат в коде.
- Ошибки API корректно обрабатываются.
- Есть журнал заявок.

## 17. Риски, которые нужно закрыть до публичного запуска

- Юридические документы не проверены.
- Курс считается неверно.
- Нет фиксации курса.
- Нет логов по заявке.
- Нет резервного копирования.
- Нет регламента AML.
- Нет защиты админки.
- Нет мониторинга ошибок.
- Нет понятного процесса возврата.
- Операторская работа держится только на переписке в Telegram.

## 18. Итог

Проект лучше делать поэтапно. Сначала нужно собрать MVP: сайт, расчет курса, заявки, Telegram-бот, оферта, карта и базовый AML-процесс. После этого можно усиливать автоматизацию, подключать полноценную админ-панель, готовить интеграцию с BestChange и расширять список направлений.

Главный фокус первой версии - не визуальная сложность, а надежность расчета курса, понятный процесс заявки, юридическая ясность и контролируемая операторская работа.
